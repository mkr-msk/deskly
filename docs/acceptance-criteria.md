# Acceptance Criteria — Deskly

## 1. Общие принципы
Acceptance Criteria описывают условия, при которых User Story считается выполненной.

Формат: Given / When / Then

---

## 2. US-1 Регистрация

### Сценарий: Успешная регистрация
- Given пользователь находится на странице регистрации
- When он вводит корректный email и пароль
- Then система создаёт нового пользователя
- And пользователь считается авторизованным

### Сценарий: Email уже используется
- Given пользователь находится на странице регистрации
- When он вводит email, который уже существует в системе
- Then система показывает ошибку
- And пользователь не создаётся

---

## 3. US-2 Вход в систему

### Сценарий: Успешный вход
- Given пользователь зарегистрирован
- When он вводит корректный email и пароль
- Then система авторизует пользователя

### Сценарий: Неверные данные
- Given пользователь находится на странице входа
- When он вводит неправильный пароль
- Then система показывает ошибку
- And доступ не предоставляется

---

## 4. US-3 Создание заявки

### Сценарий: Успешное создание
- Given пользователь авторизован
- When он заполняет title и description
- Then создаётся новая заявка со статусом OPEN
- And заявка отображается в списке пользователя

### Сценарий: Пустой заголовок
- Given пользователь авторизован
- When он пытается создать заявку без title
- Then система показывает ошибку
- And заявка не создаётся

---

## 5. US-4 Просмотр списка заявок

### Сценарий: Отображение списка
- Given пользователь авторизован
- When он открывает страницу заявок
- Then он видит список своих заявок

---

## 6. US-5 Просмотр заявки

### Сценарий: Просмотр деталей
- Given пользователь авторизован
- When он открывает заявку
- Then он видит title, description, статус и комментарии

---

## 7. US-8 Изменение статуса

### Сценарий: Успешное изменение
- Given пользователь с ролью AGENT
- And заявка существует
- When он меняет статус
- Then статус заявки обновляется

---

## 8. US-10 Добавление комментария

### Сценарий: Успешное добавление
- Given пользователь авторизован
- And заявка существует
- When он добавляет комментарий
- Then комментарий сохраняется и отображается в заявке

---

## 9. US-6 Редактирование заявки

### Сценарий: Успешное редактирование
- Given пользователь авторизован
- And заявка принадлежит ему
- When он изменяет title или description
- Then изменения сохраняются

### Сценарий: Попытка редактировать чужую заявку
- Given пользователь авторизован
- And заявка принадлежит другому пользователю
- When он пытается её редактировать
- Then система запрещает действие

---

## 10. US-7 Просмотр всех заявок

### Сценарий: Доступ исполнителя
- Given пользователь с ролью AGENT
- When он открывает список заявок
- Then он видит все заявки в системе

---

## 11. US-9 Назначение на заявку

### Сценарий: Самоназначение
- Given пользователь с ролью AGENT
- And заявка существует
- When он назначает себя исполнителем
- Then заявка получает assignedToId пользователя

---

## 12. US-11 Просмотр комментариев

### Сценарий: Отображение комментариев
- Given пользователь авторизован
- And заявка существует
- When он открывает заявку
- Then он видит список комментариев

---

## 13. US-12 Фильтрация заявок

### Сценарий: Фильтр по статусу
- Given пользователь авторизован
- When он выбирает фильтр по статусу
- Then отображаются только заявки с выбранным статусом

### Сценарий: Фильтр по приоритету
- Given пользователь авторизован
- When он выбирает фильтр по приоритету
- Then отображаются только заявки с выбранным приоритетом

---

## 14. US-13 Управление пользователями

### Сценарий: Доступ администратора
- Given пользователь с ролью ADMIN
- When он открывает список пользователей
- Then он видит всех пользователей системы

### Сценарий: Изменение роли
- Given пользователь с ролью ADMIN
- And другой пользователь существует
- When администратор изменяет его роль
- Then новая роль сохраняется
